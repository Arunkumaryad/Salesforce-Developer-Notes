# Async Apex in Salesforce | Introduction | Asynchronous Apex

## Overview

In Salesforce, Apex code can run either:

1. **Synchronously** (one step at a time while the user waits)
2. **Asynchronously** (in the background without making the user wait)

Asynchronous Apex is used when operations take a long time to complete, process large amounts of data, or need to be executed later.

---

# What is Asynchronous Apex (Async Apex)?

### Actual Definition (From the Notes)

> **An asynchronous process is a process or function that executes a task "in the background" without the user having to wait for the task to finish.**

---

## Simple Definition

Asynchronous Apex allows Salesforce to execute code in the background so users can continue working without waiting for the process to complete.

Instead of making the user wait until the code finishes execution, Salesforce places the job in a queue and executes it separately.

---

# Understanding Synchronous vs Asynchronous Processing

## Synchronous Process

In a synchronous process:

```text
Statement 1
      ↓
Statement 2
      ↓
Statement 3
      ↓
Statement 4
```

Each statement waits for the previous statement to finish.

The user must wait until the entire process completes.

---

## Asynchronous Process

In an asynchronous process:

```text
Statement 1
      ↓
Statement 2
      ↓
Call Async Process
      ↓
Statement 3
      ↓
Statement 4

                ┌────────────────────┐
                │ Async Process      │
                │ Runs in Background │
                └────────────────────┘
```

The main process continues execution immediately.

The asynchronous job runs separately in the background.

The user does not have to wait.

---

# Why Do We Need Async Apex?

In Salesforce, some operations can take a long time:

- Processing thousands of records
- Sending emails
- Calling external systems
- Data cleanup jobs
- Complex calculations
- Scheduled operations

Running these tasks synchronously may:

- Slow down users
- Cause timeout errors
- Hit governor limits

Async Apex solves these problems by executing work in the background.

---

# Advantages of Async Apex

According to the notes:

### 1. Runs in Background

The biggest advantage is that the task executes separately from the user's request.

Users can continue their work immediately.

---

### 2. Higher Governor Limits

Async Apex provides larger governor limits compared to synchronous Apex.

For example:

| Limit | Synchronous | Asynchronous |
|---------|-------------|-------------|
| SOQL Queries | 100 | 200 |
| Heap Size | 6 MB | 12 MB |

This makes Async Apex suitable for processing large amounts of data.

---

### 3. Can Be Scheduled

You can schedule Async Apex to run:

- Daily
- Weekly
- Monthly
- Specific date/time

Example:

```text
Run every day at 2 AM
```

---

### 4. Can Process More Number of Records

Async Apex is designed for large-scale processing.

Examples:

- 10,000 Accounts
- 50,000 Contacts
- Millions of records using Batch Apex

---

# Real-Life Example

Suppose a company uploads:

```text
100,000 Customer Records
```

If Salesforce processes all records synchronously:

```text
User waits
↓
System becomes slow
↓
Governor limits may hit
```

Using Async Apex:

```text
Upload Data
↓
Job Submitted
↓
User continues working
↓
Salesforce processes records in background
```

Much faster and more scalable.

# Types of Async Apex

Salesforce provides four major ways to run Apex asynchronously.

---

## 1. Future Methods

Future Methods execute Apex code asynchronously in the background.

### Syntax

```apex
@future
public static void processData() {

}
```

### Use Cases

- Callouts
- Background updates
- Non-critical processing

### Example

```apex
@future
public static void sendEmail() {

    System.debug('Email sent');

}
```

---

## 2. Batch Apex

Batch Apex is used to process a very large number of records.

Instead of processing all records at once, Salesforce processes them in small chunks called batches.

---

### Example

```text
10,000 Records
        ↓
Batch 1 → 200
Batch 2 → 200
Batch 3 → 200
```

---

### Use Cases

- Data migration
- Data cleanup
- Large-scale updates

---

### Interface

```apex
Database.Batchable
```

---

## 3. Queueable Apex

Queueable Apex is an advanced version of Future Methods.

It provides:

- Better monitoring
- Job chaining
- Complex data support

---

### Example

```apex
public class MyQueueableJob
implements Queueable {

    public void execute(
        QueueableContext qc
    ) {

        System.debug('Queueable Job');

    }
}
```

---

### Execute Queueable

```apex
System.enqueueJob(
    new MyQueueableJob()
);
```

---

## 4. Scheduled Apex

Scheduled Apex executes Apex code at a specific date and time.

---

### Example

```text
Run every day at 1 AM
```

---

### Interface

```apex
Schedulable
```

---

### Example

```apex
public class DailyJob
implements Schedulable {

    public void execute(
        SchedulableContext sc
    ) {

        System.debug(
            'Scheduled Job'
        );

    }
}
```

---

# Comparison of Async Apex Types

| Feature | Future | Queueable | Batch | Scheduled |
|----------|----------|----------|----------|----------|
| Runs Async | ✅ | ✅ | ✅ | ✅ |
| Supports Complex Objects | ❌ | ✅ | ✅ | ✅ |
| Supports Chaining | ❌ | ✅ | ✅ | ✅ |
| Processes Large Data | ❌ | ❌ | ✅ | ❌ |
| Schedule Execution | ❌ | ❌ | ❌ | ✅ |

# When to Use Which Async Apex?

### Use Future Method When

- Simple background processing
- Lightweight operations
- Small updates

---

### Use Queueable Apex When

- Need job chaining
- Need monitoring
- Passing complex objects

---

### Use Batch Apex When

- Processing thousands or millions of records
- Data migration
- Data cleanup

---

### Use Scheduled Apex When

- Daily jobs
- Weekly jobs
- Monthly jobs
- Time-based processing
