# Async Apex in Salesforce | Future Method | Asynchronous Apex

## Overview

Future Methods are one of the simplest ways to execute Apex code asynchronously in Salesforce.

A Future Method runs in the background in a separate transaction and allows Salesforce to process time-consuming tasks without making users wait.

Future Methods are commonly used for:

- Background processing
- Callouts to external systems
- Updating records asynchronously
- Performing operations that don't need an immediate response

---

# What is a Future Method?

### Actual Definition (From the Notes)

> **Runs in its own thread/transaction.**

> **Executes when system resources are available.**

---

## Simple Definition

A Future Method is an Apex method marked with the `@future` annotation that runs asynchronously in a separate transaction.

Instead of executing immediately, Salesforce places the method in a queue and executes it later when system resources become available.

---

# How Future Methods Work

### Synchronous Execution

```text
User Action
     ↓
Apex Executes
     ↓
User Waits
     ↓
Process Finishes
```

---

### Future Method Execution

```text
User Action
     ↓
Future Method Called
     ↓
Job Added To Queue
     ↓
User Continues Working

-------------------------

Background Processing
     ↓
Future Method Executes
```

The user does not need to wait for the operation to finish.

---

# How to Implement a Future Method

### Actual Note Definition

> We need to use annotation **@future** for methods.

### Syntax

```apex
@future
public static void methodName() {

}
```

---

## Example

```apex
public class AsyncApexTest {

    @future
    public static void updateLead() {

        System.debug('Future Method Executed');

    }
}
```

---

# Requirements of Future Methods

According to the notes:

---

## 1. Future Method Must Be Static

### Correct

```apex
@future
public static void updateLead() {

}
```

### Incorrect

```apex
@future
public void updateLead() {

}
```

Future methods must always be declared as `static`.

---

## 2. Must Have Void Return Type

### Correct

```apex
@future
public static void updateLead() {

}
```

### Incorrect

```apex
@future
public static String updateLead() {

}
```

Future methods cannot return values because they execute asynchronously.

---

## 3. Cannot Call a Future Method from Another Future Method

### Note Definition

> We can not call any future method (or any asynchronous process) from the future method.

### Example

```apex
@future
public static void updateLead() {

    insertLead(); // Not Allowed

}

@future
public static void insertLead() {

}
```

---

### Error

```text
Future method cannot be called from a future or batch method
```

This is exactly the error shown in the screenshots.

---

## 4. Parameters Must Be Primitive Data Types

### Allowed

```apex
Id
Integer
Long
Decimal
Double
String
Boolean
Date
Datetime
```

---

### Allowed Collections

```apex
List<String>
Set<Id>
List<Integer>
Set<String>
```

---

### Not Allowed

According to the notes:

> But not non-primitive data types (e.g. sObject, Standard or Custom Object name, Class Name).

Examples:

```apex
Lead
Account
Contact
Custom Objects
Custom Classes
```

# Example 1: Future Method to Update Lead Records

### Apex Class

```apex
public class AsyncApexTest {

    @future
    public static void updateLead() {

        List<Lead> leadList = [
            SELECT Id,
                   Name,
                   Status,
                   Company
            FROM Lead
            WHERE Company = 'Cadinal Inc.'
        ];

        List<Lead> updatedLead =
            new List<Lead>();

        for(Lead ld : leadList) {

            ld.Company = 'TCS';
            updatedLead.add(ld);

        }

        if(!updatedLead.isEmpty()) {

            update updatedLead;

        }
    }
}
```

# Why We Cannot Pass sObjects?

Suppose we try:

```apex
@future
public static void updateLead(
    List<Lead> records
) {

}
```

Salesforce throws an error.

### Error

```text
Future methods do not support parameter type of List
```

This is exactly what is shown in the screenshots.

---

# Correct Approach

Instead of passing records:

Pass record IDs.

### Correct Example

```apex
@future
public static void updateLead(
    Set<Id> recordIds
) {

    List<Lead> leadList =
    [
        SELECT Id,
               Name,
               Company
        FROM Lead
        WHERE Id IN :recordIds
    ];

}
```

---

# Execute Using Record IDs

```apex
Map<Id, Lead> newMap =
    new Map<Id, Lead>(
        [SELECT Id FROM Lead]
    );

AsyncApexTest.updateLead(
    newMap.keySet()
);
```

This is the recommended Salesforce approach.

---

```apex
public class AsyncApexTest {

    @future
```

# Limitations of Future Methods

### No Return Value

```apex
Must return void
```

---

### Static Only

```apex
Must be static
```

---

### No sObject Parameters

```apex
Lead ❌
Account ❌
Contact ❌
Custom Object ❌
```

---

### No Chaining

Cannot call:

```text
Future → Future
Future → Batch
Future → Queueable
```

---

### Execution Order Not Guaranteed

If multiple future jobs are submitted:

```text
Job A
Job B
Job C
```

Salesforce does not guarantee:

```text
A → B → C
```

They may execute in any order.

# Interview Questions

## What is a Future Method?

A Future Method is an Apex method annotated with `@future` that executes asynchronously in a separate transaction.

---

## Which Annotation Is Used For Future Methods?

```apex
@future
```

---

## Must a Future Method Be Static?

Yes.

Future methods must be static.

---

## Can a Future Method Return a Value?

No.

Future methods must have a `void` return type.

---

## Can We Pass sObjects To Future Methods?

No.

Only primitive data types or collections of primitive data types are allowed.

---

## Can One Future Method Call Another Future Method?

No.

Salesforce does not allow asynchronous processes to be called from a Future Method.

---

## What Is The Recommended Way To Pass Records?

Pass:

```apex
Set<Id>
```

Then query records inside the Future Method.

---

# Best Practices

### Pass IDs Instead of Records

```apex
Set<Id>
```

not

```apex
List<Account>
```

---

### Keep Logic Lightweight

Future methods should perform focused tasks.

---

### Avoid Multiple Future Calls

Combine operations when possible.

---

### Use Queueable Apex For New Development

Salesforce generally recommends Queueable Apex because it provides:

- Monitoring
- Chaining
- Better flexibility

# Summary

### Definition

> Runs in its own thread/transaction.

> Executes when system resources are available.

### Implementation

```apex
@future
public static void methodName() {

}
```

### Requirements

1. Must be static.
2. Must return void.
3. Cannot call another future method.
4. Parameters must be primitive types or collections of primitive types.

### Common Use Cases

- Background processing
- Callouts
- Email sending
- Record updates
- Integrations

Future Methods are the foundation of Asynchronous Apex and are often the first async feature Salesforce developers learn before moving to Queueable Apex, Batch Apex, and Scheduled Apex.
