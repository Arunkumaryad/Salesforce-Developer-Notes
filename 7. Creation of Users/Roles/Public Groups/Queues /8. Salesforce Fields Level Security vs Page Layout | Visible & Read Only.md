# Salesforce Field Level Security vs Page Layout | Visible & Read Only

## Introduction

In Salesforce, both **Page Layouts** and **Field Level Security (FLS)** are used to control how fields appear to users.

Many Salesforce beginners get confused because both options provide controls such as:

```text
Visible / Not Visible
Read Only / Editable
```

However, they work differently and have different priorities.

This guide explains:

- What is Page Layout
- What is Field Level Security
- Difference between Page Layout and FLS
- Which one has higher priority
- Real-world examples
- Interview questions

---

# Understanding the Difference

Salesforce provides two levels of control for fields:

```text
1. Page Layout
2. Field Level Security (FLS)
```

Both can affect field visibility and editability, but only one provides actual security.

---

# Page Layout

## Definition

A **Page Layout** controls how fields, buttons, related lists, and sections appear on a record page.

It is mainly used for:

- User Interface customization
- Organizing fields
- Making fields Read Only
- Making fields Required

---

## Page Layout Controls

Page Layout can control:

```text
Visible / Not Visible
Read Only / Editable
Required / Not Required
```

---

## Navigation

Navigate to:

```text
Setup
→ Object Manager
→ Teacher
→ Page Layouts
→ Teacher Layout
```

---

## Field Properties in Page Layout

Open a field from the layout.

Salesforce displays:

```text
Read Only
Required
```

### Read Only

```text
User can view the field
User cannot edit the field
```

### Required

```text
User must enter a value
before saving the record
```

Example:

```text
Teacher Name *
```

If left blank:

```text
Complete this field.
```

Salesforce displays a validation message.

---

# Field Level Security (FLS)

## Definition

Field Level Security controls whether a user can view or edit a field based on their Profile or Permission Set.

Unlike Page Layout, Field Level Security provides actual security.

---

## Field Level Security Controls

FLS can control:

```text
Visible / Not Visible
Read Only / Editable
```

---

## Navigation

Navigate to:

```text
Setup
→ Object Manager
→ Teacher
→ Fields & Relationships
→ Select Field
→ Set Field-Level Security
```

---

# Difference Between Page Layout and Field Level Security

| Feature | Page Layout | Field Level Security |
|----------|------------|----------------------|
| Visible / Hidden | ✅ | ✅ |
| Read Only | ✅ | ✅ |
| Required | ✅ | ❌ |
| Profile Based | ❌ | ✅ |
| Permission Set Based | ❌ | ✅ |
| Security Control | ❌ | ✅ |
| Higher Priority | ❌ | ✅ |

---

# Important Rule

## Field Level Security Always Wins

When Page Layout and Field Level Security conflict, Salesforce follows:

```text
Field Level Security > Page Layout
```

## Requirement

```text

```

The field should be visible only for System Administrators and hidden from all other users.

```text
Setup
→ Object Manager
→ Teacher
→ Fields & Relationships
→ New
```

```text
Number
```

Click:

```text
Next
```

Enter:

```text
Field Label : Experience
Field Name  : Experience
```

## Step 2: Configure Field Level Security

At:

```text
Step 3: Establish Field-Level Security
```

Configure:

| Profile | Visible | Read Only |
|----------|----------|-----------|
| System Administrator | ✅ | ❌ |
| Custom Sales Profile | ❌ | ❌ |

Click:

```text
Next
```

---

## Step 3: Add Field to Page Layout

Select:

```text
Teacher Layout
```

Enable:

```text
✓ Add Field
```

Click:

```text
Save
```

---

# Result

## System Administrator

Can see:

```text
Teacher Name
Dept
Teacher ID
Experience
```

---

## Custom Sales Profile

Can see:

```text
Teacher Name
Dept
Teacher ID
```

Experience field is completely hidden.

---

## Why?

Because Field Level Security is configured as:

```text
Visible = False
```

for the Custom Sales Profile.

Salesforce completely hides the field.

# Page Layout Read Only vs FLS Read Only

## Page Layout Read Only

```text
Visible
Read Only
```

Works only on the page layout.

---

## Field Level Security Read Only

```text
Visible
Read Only
```

Works everywhere:

- Record Page
- Reports
- List Views
- API
- SOQL
- Integrations

---

# Security Evaluation Order

Salesforce evaluates permissions in the following order:

```text
1. Object Permissions
2. Field Level Security
3. Page Layout
4. Record Level Access
```

---

# Interview Questions

## What is Field Level Security?

Field Level Security controls whether a user can view or edit a field through Profiles and Permission Sets.

---

## What is Page Layout?

Page Layout controls how fields appear on a record page and whether they are visible, read-only, or required.

---

## Which has higher priority?

```text
Field Level Security
```

always has higher priority than Page Layout.

---

## Can Page Layout secure data?

```text
No
```

Page Layout only controls the user interface.

---

## Can Field Level Security hide a field completely?

```text
Yes
```

The field becomes inaccessible from:

- Record Pages
- Reports
- List Views
- Search
- API
- SOQL Queries

---

# Best Practices

✔ Use Page Layout for UI customization

✔ Use Field Level Security for security

✔ Never rely only on Page Layout for data protection

✔ Use Permission Sets for additional access

✔ Follow Principle of Least Privilege

✔ Test with different Profiles before deployment

---

# Summary

Page Layout controls the appearance of fields on a record page and can make fields visible, read-only, or required. Field Level Security controls actual access to the field through Profiles and Permission Sets. When both settings conflict, Salesforce always follows Field Level Security, making it the most important mechanism for securing field data.
