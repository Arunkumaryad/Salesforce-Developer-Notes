# Use of @wire Decorator Adapter to Call Apex Class Method from LWC

## Overview

In Lightning Web Components (LWC), the **@wire** decorator is not limited to Lightning Data Service adapters like `getRecord`. It can also be used to call **Apex methods**. This approach is useful when the required data cannot be retrieved using the standard UI API or when you need to execute custom SOQL queries or business logic.

In this tutorial, you'll learn:

- What is `@wire` with Apex?
- Why do we use it?
- How to create an Apex class for LWC.
- Calling an Apex method without parameters.
- Calling an Apex method with parameters.
- Complete code explanation.
- Best practices.
- Interview questions.

---

# What is @wire with Apex?

The **@wire** decorator can be used to call an Apex method that is annotated with:

```apex
@AuraEnabled(cacheable=true)
```

The wire service automatically calls the Apex method, retrieves the returned data, and refreshes the component whenever the reactive parameters change.

Unlike imperative Apex calls, you do not need to manually invoke the method.

---

# Why Use @wire with Apex?

We use `@wire` with Apex when:

- Complex SOQL queries are required.
- Business logic needs to be executed.
- Multiple objects need to be queried together.
- UI API adapters cannot satisfy the requirement.
- Data should automatically refresh when reactive parameters change.

---

# Prerequisites

Before using `@wire` with Apex:

- The Apex method must be **static**.
- The Apex method must be **public** or **global**.
- The method must be annotated with:

```apex
@AuraEnabled(cacheable=true)
```

Without `cacheable=true`, the method cannot be used with `@wire`.

# Example 1: Calling Apex Method Without Parameters

This example retrieves all Account records from Salesforce.

---

## Apex Class

### AccountDataFetch.cls

```apex
public with sharing class AccountDataFetch {

    @AuraEnabled(cacheable = true)

    public static List<Account> getAccountList() {

        try {

            List<Account> accList = [

                SELECT Id,

                       Name,

                       Phone

                FROM Account

            ];

            return accList;

        }

        catch (Exception e) {

            throw new AuraHandledException(

                e.getMessage()

            );

        }

    }

}
```

---

## JavaScript File

### wireExampleWithApexClass.js

```javascript
import { LightningElement, wire } from 'lwc';

import getAccountList
from '@salesforce/apex/AccountDataFetch.getAccountList';

export default class WireExampleWithApexClass
extends LightningElement {

    @wire(getAccountList)

    accounts;

}
```

---

## HTML File

<template>

# How This Example Works

## Step 1

Import the Apex method.

```javascript
import getAccountList
from '@salesforce/apex/AccountDataFetch.getAccountList';
```

The syntax is:

```javascript
import methodName
from '@salesforce/apex/ClassName.MethodName';
```

---

## Step 2

Use the `@wire` decorator.

```javascript
@wire(getAccountList)

accounts;
```

The wire service automatically calls the Apex method and stores the result in the `accounts` property.

---

## Step 3

Display records.

```html
<template if:true={accounts.data}>
```

If data is returned, iterate over the records.

```html
<template
    for:each={accounts.data}
    for:item="acc">
```

Each Account object is available as `acc`.

---

## Step 4

Display the values.

```html
Name : {acc.Name}

Phone : {acc.Phone}
```

---

## Step 5

Handle errors.

```html
<template if:true={accounts.error}>

    {accounts.error}

</template>
```

# Example 2: Calling Apex Method with Parameters

Sometimes you need to pass values from LWC to Apex.

For example, retrieve only one Account using its Record Id.

---

## Updated Apex Class

```apex
public with sharing class AccountDataFetch {

    @AuraEnabled(cacheable = true)

    public static List<Account> getAccountList(Id accId) {

        try {

            List<Account> accList = [

                SELECT Id,

                       Name,

                       Phone

                FROM Account

                WHERE Id = :accId

            ];

            return accList;

        }

        catch (Exception e) {

            throw new AuraHandledException(

                e.getMessage()

            );

        }

    }

}
```

Here, the Apex method accepts an Account Id and returns only the matching record.

---

## Updated JavaScript

```javascript
import { api, LightningElement, wire } from 'lwc';

import getAccountList
from '@salesforce/apex/AccountDataFetch.getAccountList';

export default class WireExampleWithApexClass
extends LightningElement {

    @api recordId;

    @wire(getAccountList, {

        accId: '$recordId'

    })

    accounts;

}
```

---

# How Parameter Passing Works

The wire configuration:

```javascript
@wire(getAccountList, {

    accId: '$recordId'

})
```

maps like this:

```
recordId (LWC)

↓

accId

↓

Apex Method

↓

getAccountList(Id accId)
```

Because `$recordId` is reactive, whenever the current record changes, the Apex method runs again automatically.

---

# Understanding the '$' Symbol

Notice:

```javascript
accId: '$recordId'
```

The `$` makes the parameter **reactive**.

If the value of `recordId` changes, the wire service automatically calls the Apex method again with the new value.

Without `$`, the value would be treated as a static string.

---

# Difference Between UI API and Apex Wire

| UI API Adapter | Apex Wire |
|---------------|-----------|
| Uses built-in Salesforce adapters | Uses custom Apex methods |
| Limited to supported operations | Can execute any custom business logic |
| No SOQL required | Uses SOQL queries |
| Better for standard record operations | Better for complex requirements |

- Automatic server call.
- Reactive data updates.
- Supports custom SOQL.
- Supports custom business logic.
- Cleaner code.
- Built-in caching.
- Less manual JavaScript.

# Common Mistakes

## Forgetting cacheable=true

Wrong

```apex
@AuraEnabled
```

Correct

```apex
@AuraEnabled(cacheable=true)
```

---

## Method is Not Static

Wrong

```apex
public List<Account> getAccountList()
```

Correct

```apex
public static List<Account> getAccountList()
```

---

## Forgetting to Import the Apex Method

Wrong

```javascript
@wire(getAccountList)
```

Correct

```javascript
import getAccountList

from '@salesforce/apex/AccountDataFetch.getAccountList';
```

---

## Using Wrong Parameter Name

If Apex method is:

```apex
getAccountList(Id accId)
```

Use:

```javascript
@wire(getAccountList, {

    accId: '$recordId'

})
```

The JavaScript property name must exactly match the Apex parameter name.

---

## Forgetting Error Handling

Always check:

```html
accounts.data
```

and

```html
accounts.error
```

---

# Best Practices

- Always use `@AuraEnabled(cacheable=true)` for wired Apex methods.
- Keep Apex methods lightweight.
- Return only the fields you need.
- Handle both success and error states.
- Use reactive parameters (`$`) whenever data depends on another property.
- Use imperative Apex instead of `@wire` when data should be fetched only after a user action (such as clicking a button).

# Interview Questions

### What is required to call an Apex method using @wire?

The Apex method must be:

- `public` or `global`
- `static`
- Annotated with `@AuraEnabled(cacheable=true)`

---

### Why is cacheable=true mandatory?

Because the Wire Service only works with cacheable Apex methods. It enables client-side caching and improves performance.

---

### Can @wire call non-static Apex methods?

No. The Apex method must be static.

---

### Why do we use `$recordId`?

The `$` makes the parameter reactive. When `recordId` changes, the wire service automatically invokes the Apex method again.

---

### What happens if the Apex method throws an exception?

The error is available in:

```javascript
accounts.error
```

You should handle it appropriately in your component.

---

### When should we use @wire instead of imperative Apex?

Use `@wire` when:

- Data should load automatically.
- The Apex method is cacheable.
- You want reactive updates.

Use imperative Apex when:

- Data should load only after a user action.
- You need DML operations (insert, update, delete).
- The method cannot be cacheable.
